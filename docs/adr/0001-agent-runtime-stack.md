# ADR-0001：Agent Runtime 技术选型（Week 1）
状态：Accepted  
日期：2026-04-20

## 背景
- TokenFlow 需要一个可本地部署的第三端执行环境，承载节点执行、模型推理与后续 RAG 流程。
- 当前阶段优先目标是先打通 `DAG -> Runtime -> 结构化结果` 的最小闭环，再逐步增强吞吐与安全隔离。
- 约束条件：
  - 开发与测试环境以单机为主。
  - 需要支持离线/弱网部署。
  - 前后端都已经基于 FastAPI + Python 生态。

## 决策
采用以下组合：
- 推理层：`llama-cpp-python`
- 模型格式：`GGUF`
- 对外运行时接口：`FastAPI`

## 选型理由
1. `llama-cpp-python`
- Python 侧易于与现有服务集成，调用成本低。
- 支持 CPU 场景，可在无 GPU 的开发机上先行验证。
- 与 GGUF 配套成熟，便于本地模型文件管理。

2. `GGUF`
- 面向本地推理分发友好，模型文件管理简单。
- 支持量化版本（如 4bit/8bit）以降低显存/内存压力。
- 便于在 `models/` 目录实施白名单与审计策略。

3. `FastAPI`
- 现有后端已采用 FastAPI，团队维护成本最低。
- 能直接产出 OpenAPI 文档，便于前端联调与协议冻结。
- 适合快速扩展健康检查、限流、超时等中间能力。

## Redis 队列评估（是否必须）
结论：**Week 1 不强制引入 Redis**，先以进程内并发控制 + 超时机制满足最小链路。

不立即引入的原因：
- 当前目标是协议与执行骨架冻结，优先降低系统复杂度。
- 单机场景下，队列带来的收益低于维护成本（额外部署、可用性与监控负担）。

触发引入 Redis 的阈值（满足任一即可立项）：
- 平均并发任务数持续 `> 8` 且高峰排队时间 `> 2s`。
- 需要跨进程/跨实例的任务可见性与重试机制。
- 需要异步长任务状态持久化（重启后恢复）与削峰填谷。

## 硬件与性能基线（验收目标）
说明：以下为 **Week 1/2 阶段基线目标**，用于容量预估与后续验证，不代表最终上线 SLA。

| 模型级别（GGUF） | 建议内存/显存 | 并发（单实例） | 目标 P95 延迟 | 目标吞吐 |
| --- | --- | --- | --- | --- |
| 0.5B ~ 1.5B（Q4） | RAM 8GB+ | 4 | <= 1.5s（短文本） | 2~4 req/s |
| 3B ~ 7B（Q4/Q5） | RAM 16GB+ 或 VRAM 8GB+ | 2 | <= 3.5s（短文本） | 0.8~2 req/s |
| 8B ~ 14B（Q4） | RAM 32GB+ 或 VRAM 16GB+ | 1~2 | <= 6.0s（短文本） | 0.3~1 req/s |

通用约束建议：
- Runtime 默认超时：`30s`（可按节点覆盖）。
- 单请求最大源码：`<= 200KB`。
- 单请求资源总量：`<= 10MB`（文本等价后）。
- 并发上限先设为 `CPU 核数 / 2`，再根据压测结果调优。

## 影响与后续
- 本 ADR 支撑 Week 1 “协议与骨架冻结”里程碑，可直接进入 Week 2 runtime API 落地。
- Week 2/3 需要补充：
  - 基于真实模型与真实 prompt 的压测报告。
  - 是否启用 Redis 的复评结果。
  - 安全隔离策略（危险导入限制、日志脱敏、资源上限）细化清单。
